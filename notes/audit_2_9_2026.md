# Backyard Code Audit — February 9, 2026

Scope: full codebase review covering security, code quality, and architectural concerns.

> **Status: All 22 findings remediated.**

---

## Critical

### 1. SQL Injection in `getAuthorFeed` (`src/lib/server/feed.ts`, lines ~330–360)

The `vd` variable (derived from `viewerDid` or `authorDid`) is interpolated directly into SQL via template literal string rather than using a parameterized placeholder:

```typescript
const vd = viewerDid || authorDid;
const query = `
    ...
    (SELECT uri FROM likes WHERE author_did = '${vd}' AND subject_uri = p.uri LIMIT 1) as viewer_like,
    (SELECT uri FROM reblogs WHERE author_did = '${vd}' AND subject_uri = p.uri LIMIT 1) as viewer_reblog
    ...
`;
```

Although DIDs are *normally* alphanumeric identifiers, the firehose ingests data from the entire AT Protocol network. A crafted DID containing SQL metacharacters (e.g. from a malicious PDS) could reach this code path since `authorDid` is resolved from the `profiles` table, which is populated by the firehose. This is an exploitable SQL injection.

**Recommendation:** Replace the string interpolation with a numbered parameter (`$3`) and pass `vd` through the `params` array, matching the pattern already used in `getTimeline`.

### 2. No Input-Size Guardrails on Firehose-Ingested Data (`src/lib/server/firehose.ts`)

The `indexRecord` function accepts arbitrary record data from the Jetstream firehose and inserts it into PostgreSQL without any schema validation or size limits:

- `r.text` — no length check (could be megabytes)
- `r.facets`, `r.media` — serialized to JSON with no depth or size cap
- `r.tags` — no count or individual-length limit
- `r.createdAt` — no format validation (invalid timestamps will cause query errors)

An adversarial actor on the AT Protocol network could publish records with extremely large payloads targeting these columns, leading to storage exhaustion or degraded query performance.

**Recommendation:** Validate incoming records against the lexicon schemas (or at minimum enforce maximum lengths/sizes) before insertion. Truncate or reject oversized fields.

## High

### 3. Hardcoded Salt in Session Key Derivation (`src/lib/server/session.ts`, line 9)

```typescript
return crypto.scryptSync(secret, 'backyard-salt', 32);
```

The static salt `'backyard-salt'` means any two deployments with the same `SESSION_SECRET` produce identical encryption keys. If the default secret leaks (it's already in the source), every default deployment shares keys.

**Recommendation:** Derive the salt from the deployment's `PUBLIC_URL` or another instance-unique value, or generate a random salt and persist it alongside the database.

### 4. Insecure Default `SESSION_SECRET` (`docker-compose.yml`, `src/lib/server/session.ts`)

The docker-compose default is `change-me-to-a-random-secret-at-least-32-characters-long!!`, and `session.ts` falls back to `development-secret-must-be-at-least-32-characters-long!!`. Neither triggers a warning log in production. If `SESSION_SECRET` is unset, sessions are trivially forgeable by anyone who reads the source.

**Recommendation:** Refuse to start (or at minimum log a loud warning) when `NODE_ENV=production` and `SESSION_SECRET` is absent or matches the known default. Consider generating a random secret on first boot and persisting it.

### 5. No Rate Limiting on Any Endpoint

There is no rate limiting anywhere — not on OAuth login initiation (`/api/auth/login`), post creation, likes, follows, comments, or search. This exposes the application to:

- Spam creation of posts/comments/reblogs
- Abuse of the OAuth authorize flow (which makes requests upstream to the user's PDS)
- Search abuse (each query hits `ILIKE %...%` against the full profiles table)

**Recommendation:** Implement per-IP or per-session rate limiting, at minimum on write endpoints and authentication. A reverse proxy (nginx, Caddy) rate-limit configuration would also help.

### 6. Missing `request.json()` Error Handling on All API Routes

Every API endpoint calls `await request.json()` outside a try/catch. A request with an invalid or empty body produces an unhandled exception and a generic 500 response rather than a descriptive 400.

Affected files:
- `src/routes/api/auth/login/+server.ts`
- `src/routes/api/post/+server.ts`
- `src/routes/api/like/+server.ts`
- `src/routes/api/follow/+server.ts`
- `src/routes/api/reply/+server.ts`
- `src/routes/api/repost/+server.ts`

**Recommendation:** Wrap `request.json()` in try/catch and return a 400 with an error message on parse failure.

## Medium

### 7. No Security Headers (CSP, etc.)

The application sets no `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, or `Strict-Transport-Security` headers. While Svelte's default escaping provides baseline XSS protection, CSP adds defence-in-depth.

**Recommendation:** Add security headers in `hooks.server.ts`, e.g.:
```typescript
response.headers.set('X-Content-Type-Options', 'nosniff');
response.headers.set('X-Frame-Options', 'DENY');
response.headers.set('Referrer-Policy', 'strict-origin-when-cross-origin');
```
Add a CSP once the full set of required external origins is known.

### 8. Race Condition in Database Initialization (`src/hooks.server.ts`)

```typescript
let dbInitialized = false;
// ...
if (!dbInitialized) {
    await initializeDatabase();
    dbInitialized = true;
```

Multiple concurrent requests at startup can trigger `initializeDatabase()` in parallel. While the SQL is mostly idempotent (`CREATE TABLE IF NOT EXISTS`), the `UPDATE reblogs SET root_post_uri = ...` migration could execute redundantly and the `DELETE FROM oauth_state ...` cleanup could conflict.

**Recommendation:** Use a `Promise`-based singleton pattern so only the first caller executes initialization and subsequent callers await the same promise.

### 9. Race Condition in OAuth Client Initialization (`src/lib/server/oauth.ts`)

```typescript
let oauthClient: NodeOAuthClient | null = null;
export async function getOAuthClient() {
    if (oauthClient) return oauthClient;
    // async work...
    oauthClient = new NodeOAuthClient({...});
```

Same issue: concurrent calls before the singleton is set will each create a new `NodeOAuthClient`, potentially generating multiple ephemeral keys in development or causing other side effects.

**Recommendation:** Cache the initialization promise:
```typescript
let clientPromise: Promise<NodeOAuthClient> | null = null;
export function getOAuthClient() {
    if (!clientPromise) clientPromise = createOAuthClient();
    return clientPromise;
}
```

### 10. LIKE Pattern Injection in Search (`src/lib/server/identity.ts`, `searchProfiles`)

User-supplied search queries are passed into `ILIKE` patterns via `%${query}%`. Special SQL LIKE metacharacters (`%`, `_`) in user input will alter matching behavior. For example, searching for `%` matches every profile.

**Recommendation:** Escape LIKE metacharacters in user input before interpolation:
```typescript
const escaped = query.replace(/[%_\\]/g, '\\$&');
// then use `%${escaped}%` with ILIKE ... ESCAPE '\\'
```

### 11. `did:web` Domain Not Validated (`src/lib/server/identity.ts`, line 24)

```typescript
const domain = did.slice('did:web:'.length);
const res = await fetch(`https://${domain}/.well-known/did.json`);
```

No validation ensures `domain` is a legitimate hostname. A DID like `did:web:169.254.169.254` could trigger SSRF to cloud metadata endpoints. Domains with encoded path separators could redirect to internal services.

**Recommendation:** Validate that the domain is a well-formed public hostname. Reject IP addresses, localhost, and private ranges.

### 12. Missing Input Validation on Multiple API Endpoints

- `/api/like` — no validation of `uri`, `cid`, or `liked` fields
- `/api/follow` — no DID format validation
- `/api/repost` — no validation of `uri` or `cid`
- `/api/reply` — no text length limit, no `subjectCid`/`rootCid` format validation

**Recommendation:** Validate AT-URI format (`at://did/collection/rkey`), CID format, DID format, and impose text length limits consistently across all endpoints.

## Low

### 13. OAuth State Cleanup Only Runs at Startup

```sql
DELETE FROM oauth_state WHERE created_at < NOW() - INTERVAL '1 hour';
```

This runs once during `initializeDatabase()`. Abandoned states accumulate until the next application restart.

**Recommendation:** Add a periodic cleanup job (e.g., `setInterval` every hour) or add TTL-based cleanup on reads.

### 14. Profile Cache Has No Expiration Policy

The `profiles` table has `updated_at` but no TTL. `ensureProfile` only re-resolves when both `display_name` and `avatar` are null. A user who changes their display name or description on their PDS will see stale data indefinitely in Backyard.

**Recommendation:** Re-resolve profiles periodically (e.g., if `updated_at` is older than 24 hours) or when requested via a refresh button.

### 15. Recursive CTE Depth Limit of 50 May Be Too High (`src/lib/server/feed.ts`)

The `buildReblogChains` function uses `WHERE c.depth < 50`. A malicious user could create a 50-deep reblog chain, causing expensive recursive queries for every feed load that includes it.

**Recommendation:** Lower the depth limit to a more practical value (e.g., 10–15) and/or cache chain results.

### 16. Database Pool Uses Default Configuration (`src/lib/server/db.ts`)

The `Pool` is created without explicit connection limits, timeouts, or error handlers. The default `pg` pool allows 10 connections with no idle timeout. Under load, requests will queue without feedback.

**Recommendation:** Configure `max`, `idleTimeoutMillis`, `connectionTimeoutMillis`, and attach a `pool.on('error', ...)` handler for unexpected disconnections.

## Code Quality

### 17. Duplicated Profile-Row-to-Object Mapping

The pattern of mapping a database row to a `BackyardProfile` is repeated four times:
- `identity.ts` → `ensureProfile()` (line ~65)
- `identity.ts` → `getProfileByHandle()` (line ~201)
- `identity.ts` → `searchProfiles()` (line ~228)
- `feed.ts` → `resolveProfiles()` (line ~20)

**Recommendation:** Extract a `mapRowToProfile(row): BackyardProfile` helper and import it everywhere.

### 18. Duplicated Root-Post-URI Resolution Logic

The logic to resolve `root_post_uri` by checking if the parent subject is itself a reblog appears in both:
- `repo.ts` → `createReblog()` (line ~150)
- `firehose.ts` → `indexRecord()` case `NSID.REBLOG` (line ~130)

**Recommendation:** Extract to a shared function like `resolveRootPostUri(subjectUri): Promise<string>`.

### 19. Unused Dependencies in `package.json`

- `iron-session` — imported nowhere; session management uses custom `crypto` implementation
- `@atproto/sync` — imported nowhere; the firehose uses raw WebSocket via Jetstream

**Recommendation:** Remove both packages to reduce install footprint and audit surface.

### 20. Unused Imports in `src/routes/compose/+page.svelte`

```typescript
import type { PageData } from './$types.js';
import type { LayoutData } from '../$types.js';
```

Neither `PageData` nor `LayoutData` is used in the component.

**Recommendation:** Remove the unused imports.

### 21. Rich Text Facets Not Rendered in Frontend

Posts, comments, and reblog text contain `facets` (AT Protocol rich text spans for mentions, links, etc.) generated via `RichText.detectFacets()`, but the frontend renders all text as plain `<p>{text}</p>`. Detected links and mentions are stored in the database but never displayed as clickable elements.

**Recommendation:** Implement a facet renderer component that converts AT Protocol facet spans into clickable `<a>` tags. Take care to sanitize facet byte offsets and ensure they don't produce XSS vectors.

### 22. Logout Uses Form Action Without Page Load Guard

`/logout/+page.server.ts` defines a form `action` but no `load` function. The `+page.svelte` says "signing out..." but there's no auto-submission — the user must trigger the form action via another mechanism (e.g., the header's form button posts directly to `/logout`). The `+page.svelte` is effectively dead code since users never see it.

**Recommendation:** Either remove the svelte page and convert to a standalone `+server.ts` endpoint, or add auto-submission logic in the page.
